# 调研指南（联网与记录）

## 何时必须考虑调研

- 竞品功能、价格、定位表述 **不确定**
- Apple **审核、隐私、账号登录、内购** 等政策或实践需核对
- **技术方案**（如后台任务、HealthKit、相机/麦克风权限等）存在多种说法需交叉验证
- 市场体量、惯例、benchmark **无内部数据** 且影响决策

若用户明确「不要联网」，在文档中标注 **假设** 与 **待核实**。

## 执行步骤

1. **写清调研问题**（1–3 条可验证的子问题）
2. **检索**：优先官网、Apple Developer、App Store Review Guidelines、WWDC / 技术文档；其次权威媒体与文档日期较新的二手总结
3. **交叉验证**：单一博客或论坛帖不单独作为强依据
4. **记录**：写入 `docs/research/YYYYMMDD-主题.md`（见下方模板）
5. **回写**：把「对本产品的含义」写入对应 `docs/0x-*.md`，避免 research 与正式文档重复全文

## 竞品分析框架（简版）

1. 选 3–5 个竞品（直接优先）
2. 维度：定位、核心 loop、功能列表、付费点、评价痛点、平台与版本要求
3. 输出：**对比矩阵**（进 `docs/03-竞品分析.md`）+ **原始链接与备注**（进 `docs/research/`）

## Persona / 用户假设

- 列出可证伪假设（例如「每周训练 ≥2 次」）
- 标注验证方式（访谈、问卷、上架后数据）
- 验证前在 `docs/02-用户研究.md` 标明 **假设** 字样

## 技术可行性评估维度

- 官方 API 能力与最低系统版本
- 权限与隐私声明成本
- 离线/弱网、耗电、包体积
- 第三方 SLA 与数据出境（若涉及）

## 质量检查清单

- [ ] 每条重要结论至少对应一个可追溯来源
- [ ] 标注信息日期（政策与商店规则易变）
- [ ] 区分「事实」与「建议/推测」
- [ ] 已更新 `docs/00-讨论记录.md` 中「调研结论已采纳」说明（若影响决策）
